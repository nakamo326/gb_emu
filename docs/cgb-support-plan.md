# ゲームボーイカラー (CGB) 対応 実装計画

## ステータス

ドラフト（2026-07-01 時点）。実装着手前の設計計画。未決事項は順次確定していく。

## 目的とスコープ

現状 DMG（初代ゲームボーイ）専用の本エミュレータを、ゲームボーイカラー (CGB) 対応
にする。到達目標は以下。

- CGB 専用 / CGB 対応 ROM をカラーで実行できる。
- 既存の DMG ROM も従来どおり（あるいは CGB 側の DMG 互換モードで）動作する。
- `host`（SDL2）と `teensy`（組み込み）の両プラットフォームでカラー表示できる。

スコープ外（初期リリースでは扱わない）:

- 赤外線通信 (0xFF56 RP)
- シリアル通信（現状も未実装）
- 完全なサイクル精度の FIFO ピクセルパイプライン（現状のスキャンライン方式を踏襲）

## 現状アーキテクチャの要約（調査結果）

ワークスペースは `core`（プラットフォーム非依存のエミュレータ本体）/ `host`（SDL2）/
`teensy`（組み込み）の 3 クレート構成。ルート `Cargo.toml` の workspace members は
`core` と `host` のみで、`teensy` は別ターゲット（`thumbv7em`）向けに単独ビルドする。

- **`core`**: `#![no_std]` 属性は明示されていないが `std::` を一切使わず、`heapless`
  のみに依存。実質 no_std 互換で teensy から利用されている。DMG/CGB を型で分けておらず、
  実行時状態を持つ単一の `Ppu` / `Mmu` / `Cpu`。
- **CPU** (`core/src/cpu.rs`): `apply_dmg_init()` で DMG の初期レジスタ（A=0x01 等）を
  設定。ダブルスピードの概念なし。
- **MMU** (`core/src/mmu.rs`): アドレスデコードして各コンポーネントへ委譲。CGB レジスタは
  未対応（詳細は後述のギャップ表）。
- **PPU** (`core/src/ppu.rs`): VRAM 1 バンク固定 (`vram: [u8; 0x2000]`)、スキャンライン方式。
  `buffer: [u8; 160*144]` にパレットインデックス 0–3 を書き、`pixel_buffer() -> &[u8]` で
  公開。BGP/OBP0/OBP1 の DMG パレットのみ。
- **Display トレイト** (`core/src/platform.rs`): `draw(&mut self, buffer: &[u8])` で
  0–3 のインデックス配列を受け取る。
  - `host`: `host/src/lcd.rs` の `SdlDisplay::draw` が 4 色の固定 RGB24 パレットへ変換して
    SDL テクスチャ更新。
  - `teensy`: `teensy/src/display/mod.rs` の `DmaDisplay::draw` が `PALETTE: [u16; 4]`
    （RGB565）へ変換し、DMA で SPI パネルへ転送。
- **カートリッジ / MBC**:
  - `host` (`host/src/cartridge.rs`): `MemoryBankController` トレイトで `RomOnly` /
    `Mbc1` のみ実装。それ以外は `RomOnly` にフォールバック。ROM ヘッダ 0x0147 でタイプ判定。
  - `teensy` (`teensy/src/cartridge.rs`): SD カードではなく **実 GB カートリッジを GPIO
    バスで直接読む** `GpioCart`。MBC は実カート側のチップが担う。配線コメントに「GB カートは
    DMG・GBC ともに 5V 系」と明記されており、電気的には GBC カートも読める設計。
- **メインループ** (`host/src/main.rs`): `gb.step()`（1 M-cycle）を呼び、ナノ秒精度で
  スピンウェイトしてタイミングを刻む。CPU_CLOCK_HZ = 4.19MHz 固定。

## CGB 機能ギャップ分析（コンポーネント別）

### MMU（未対応 CGB I/O レジスタ）

| アドレス | レジスタ | 用途 | 現状の扱い |
|---|---|---|---|
| 0xFF4D | KEY1 | CPU ダブルスピード切替 | `read` で `0xFF` 固定、`write` は無視 |
| 0xFF4F | VBK | VRAM バンク選択 (0/1) | 未デコード → PPU に届かない |
| 0xFF51–55 | HDMA1–5 | 汎用 DMA / HBlank DMA | 未デコード |
| 0xFF56 | RP | 赤外線（スコープ外） | 未デコード |
| 0xFF68–69 | BCPS/BCPD | BG カラーパレット | 未デコード |
| 0xFF6A–6B | OCPS/OCPD | OBJ カラーパレット | 未デコード |
| 0xFF6C | OPRI | OBJ 優先度モード (OAM 順 / X 座標順) | 未デコード |
| 0xFF70 | SVBK | WRAM バンク選択 (1–7) | 未デコード |

### WRAM

`core/src/wram.rs` は `ram: [u8; 0x2000]`（8KB 固定）。CGB は 32KB（バンク 0 固定 +
バンク 1–7 を SVBK で 0xD000–0xDFFF に切替）。

### CPU / タイミング

- ダブルスピード（KEY1）未対応。ダブルスピード時は CPU・タイマー・DMA・シリアルが 2 倍速、
  **PPU と APU は通常速度**を維持する必要がある。
- `apply_dmg_init` に相当する CGB 初期レジスタ設定（A=0x11 等）が未実装。
- メインループのタイミング（`host/src/main.rs`）は 1 M-cycle = 固定ナノ秒。ダブルスピード
  時の刻み幅切替が必要。

### PPU（最大の作業領域）

1. **VRAM 2 バンク化**: `vram: [u8; 0x2000]` → `[u8; 0x4000]`（or 2 面）。VBK で選択。
   バンク 1 には BG マップのタイル属性（0x9800–0x9FFF に対応する領域）に加え、
   タイルデータ（0x8000–0x97FF）も置ける（タイル属性 bit3 で参照バンクを選択）。
2. **BG タイル属性**: 各タイルにパレット番号 (0–7)、タイルデータの VRAM バンク、
   水平/垂直反転、BG-to-OBJ 優先度ビット。`get_tile_idx_from_tile_map` /
   `render_bg` / `render_window` の全面改修。
3. **CGB カラーパレット**: BG 8 パレット + OBJ 8 パレット、各 4 色 × RGB555。
   BCPS/BCPD・OCPS/OCPD による読み書き（オートインクリメント付き）。
4. **OBJ の CGB 属性**: スプライト flags の下位 3 ビットで CGB パレット選択、bit3 で
   VRAM バンク選択。`render_sprites` の改修。あわせて **OBJ-to-OBJ 優先度規則**が変わる:
   CGB（OPRI=0）では X 座標に関係なく OAM インデックスの小さいスプライトが前面に来る。
   現状の `render_sprites` は X 座標を第一鍵にソートしている（DMG 挙動）ため、
   OPRI に応じてソート鍵を切り替える必要がある。
5. **優先度規則の変更**: CGB では LCDC bit0 が「BG/Window の master priority」の意味に
   変わり、BG 属性の優先度ビットと OAM 優先度ビットの複合ルールになる。
6. **出力フォーマット**: `buffer: [u8; N]`（0–3 インデックス）では色を表現できない。
   カラー化した実 RGB を出力する必要がある（設計方針で詳述）。

### Display トレイトと各プラットフォーム

`draw(&[u8])`（0–3 インデックス）前提。カラー出力に変えると host / teensy 両実装の
色変換部（`host/src/lcd.rs` の 4 色マッチ、`teensy/src/display/mod.rs` の `PALETTE` と
`fill_buffer`）が波及する。

### カートリッジ / MBC

- `host` は MBC1 まで。CGB ゲームが多用する **MBC3（RTC 含む）・MBC5 が未対応**。
  `host/src/cartridge.rs` のタイプ判定に分岐追加が必要。
- CGB フラグ（ROM 0x0143: 0x80=CGB 対応 / 0xC0=CGB 専用）の読み取りが現状どこにもない。
- `teensy` は実カート依存なので MBC 実装は不要。ただし CGB 判定用に 0x0143 を読む口が必要。

## 設計方針の決定

### 決定 1: DMG/CGB は「実行時フラグ」で切り替える（型分けしない）

`core` に機種を表す状態（例: `Model { Dmg, Cgb }` あるいは `cgb_mode: bool`）を `Mmu` /
`Ppu` が保持し、ROM ヘッダ 0x0143 とブートロム有無から起動時に決定する。

- 理由: `Ppu<Dmg>` / `Ppu<Cgb>` のような型パラメータ化は `GameBoy<C,D,A,I>` の
  ジェネリクスをさらに増やし、no_std / teensy 側の記述も膨らむ。CGB は「DMG のスーパーセット」
  であり、分岐は限定的（レジスタ有効化・パレット参照・優先度）なので実行時フラグで十分。
- DMG モードでは CGB レジスタを無効化し、従来どおりの挙動を保つ。

### 決定 2: PPU 出力を RGB555（`u16`）に統一する

`pixel_buffer` を `[u8]`（インデックス）から `[u16]`（RGB555 カラー）へ変更する。
DMG モードでも BGP/OBP を適用した結果を疑似カラー（従来の 4 色を RGB555 化）に変換して
同じ経路で出力する。

- 理由: これが CGB 対応で最も波及の大きい決定。インデックス配列のままでは CGB の
  32768 色を表現できない。PPU 内でカラー確定まで行い、Display 層は「RGB を並べて転送するだけ」
  にすると host / teensy 双方の実装が単純化し、DMG/CGB の分岐が Display 層から消える。
- Display トレイトを `draw(&mut self, buffer: &[u16])` に変更。
  - `host`: RGB555 → RGB24 変換に置換（現状の 4 色マッチを撤去）。
  - `teensy`: RGB555 → RGB565 変換に置換（`PALETTE` 参照廃止、`fill_buffer` を色変換に）。
    フレームバッファは既に RGB565 の `[u32; 11520]` を DMA 転送しており、構造は流用できる。
- 代替案（不採用）: DMG は `[u8]`・CGB は別メソッド、の二経路。Display 実装が汚れ、
  分岐が各プラットフォームに漏れるため退けた。

### 決定 3: メモリ予算（teensy）

- VRAM 8KB→16KB、WRAM 8KB→32KB、カラーパレット 128B、PPU の出力バッファが
  `[u8; 23040]`→`[u16; 23040]`（+23KB）。`bg_pixel_buffer` は優先度判定用に別途維持。
  なお teensy の表示側 `FB`（二重バッファ `[[u32; 11520]; 2]`、RGB565 を 2px/u32 でパック）は
  既にカラー幅を持つためサイズは変わらない。
- i.MX RT1062 (Teensy 4.1) は 1MB のオンチップ RAM を持つ（512KB の FlexRAM を ITCM/DTCM に
  配分し、別途 512KB の OCRAM2）。現構成では DTCM を 320KB 確保している（`teensy/build.rs`）。
  増分（合計 +60KB 程度）は収まる見込み。ただし `Ppu` 構造体の肥大化で静的確保・スタック配置に
  注意（表示側 `FB` は `static mut`）。
- パフォーマンス: 属性参照・パレット変換でスキャンライン処理が重くなる。既存の FPS /
  フレーム負荷オーバーレイ（`record_work` / `update_fps`）で退行を計測しながら進める。

### 決定 4: DMG ROM の表示は「従来の緑4色を維持」する

CGB 対応後も、DMG ROM（および CGB モードで動く DMG 互換 ROM）は現状の DMG グリーン
4 色パレットで表示する。CGB 実機のブートロムが行う自動着色（互換パレット）は初期リリース
では実装しない。

- 理由:
  1. **実機の自動着色は再現コストが高い**。CGB ブートロムは ROM タイトルのハッシュから
     内蔵テーブル（数十種のプリセット配色）を引いて着色する複雑な挙動で、忠実な再現には
     CGB ブートロム（0x900 バイト）か着色テーブルの移植が必要。CGB 本来のカラー対応という
     主目的から外れ、費用対効果が低い。
  2. **既存ユーザーの期待を壊さない**。本エミュはこれまで DMG グリーン 4 色で表示してきた
     ため、DMG ROM の見た目を維持する方が自然で、フェーズ 1 の回帰確認（見た目が変わらない
     こと）とも整合する。
  3. **他エミュの一般的な既定**とも一致する。多くの実装はデフォルトを単純なグレー/緑
     パレットとし、自動着色や任意配色はオプション扱いにしている。
- 位置づけ: CGB 自動着色は将来のオプション機能（設定で切替）として残す。決定 2 で出力は
  RGB555 に統一するため、DMG の 4 色を別の配色（グレースケール等）に差し替えるのは
  パレット定数の変更のみで可能。拡張の余地は確保される。

### 未決事項（実装前にユーザー確認したい）

1. ~~DMG ROM の表示~~ → **決定 4 で確定（従来の緑4色を維持）**。
2. **teensy でのカラー実行の優先度**: 実カート経由で CGB ROM を回すところまで初期スコープに
   含めるか、まずは host のカラー化を優先するか。
3. **ダブルスピード対応の要否**: 多くの CGB ゲームで使われるため入れる前提だが、
   初期リリースでは通常速度のみとし後続フェーズに回す選択肢もある。

## 実装フェーズ

各フェーズは単体でビルド・動作確認できる粒度に分け、コミットは意味のある最小単位で行う。

### フェーズ 0: 基盤（機種フラグと CGB 判定）

- ROM ヘッダ 0x0143 を読む口を用意し、`core` に `cgb_mode` フラグを導入。
- `Mmu::new` / `GameBoy::new` に機種を伝播。`apply_cgb_init`（CPU: A=0x11 等）を追加。
- この段階では挙動は DMG 相当のまま（フラグを持つだけ）。

### フェーズ 1: Display 出力の RGB555 化（DMG 互換のまま）

- `Ppu` の出力を `[u16]`（RGB555）に変更。DMG パレットを RGB555 に変換して出力（決定 4 に
  従い従来の緑 4 色を RGB555 化）。
- Display トレイトを `draw(&[u16])` に変更し、host / teensy の色変換を更新。
- **既存 DMG ROM が従来どおり表示されることを回帰確認**（見た目が変わらないこと）。
- この時点で CGB 色はまだ無いが、パイプラインがカラー対応になる。

### フェーズ 2: WRAM / VRAM バンクと CGB レジスタ

- WRAM 32KB 化 + SVBK (0xFF70)。
- VRAM 16KB 化 + VBK (0xFF4F)。
- CGB カラーパレット RAM と BCPS/BCPD・OCPS/OCPD (0xFF68–6B)、OPRI (0xFF6C) の read/write。
- MMU のアドレスデコードにこれらを追加。まだ描画には使わない（レジスタの箱を用意）。

### フェーズ 3: CGB BG / Window 描画

- BG タイル属性（VRAM バンク 1）を参照し、パレット・反転・バンク・優先度を反映。
- CGB BG パレットで着色。LCDC bit0 の master priority 規則を実装。
- CGB 専用 ROM のタイトル画面がカラーで出ることを確認。

### フェーズ 4: CGB スプライト描画

- OAM flags の CGB パレット (bit0–2) / VRAM バンク (bit3) を反映。
- BG-to-OBJ / OBJ-to-OBJ 優先度の複合規則を実装。OPRI (0xFF6C) に応じて OBJ-to-OBJ 優先度を
  OAM インデックス順（CGB 既定）と X 座標順（DMG 互換）で切り替える。

### フェーズ 5: MBC3 / MBC5 対応（host）

- `host/src/cartridge.rs` に MBC3（RTC は最小実装 or 後回し）と MBC5 を追加。
  0x0147 判定を拡張。
- teensy は実カート依存のため対象外。

### フェーズ 6: ダブルスピード（KEY1）

- KEY1 の read/write と `stop` 命令によるスピード切替を実装。
- CPU/タイマー/DMA を倍速、PPU/APU を通常速度に保つよう `GameBoy::step` とメインループの
  タイミングを調整。

### フェーズ 7: HDMA（汎用 DMA / HBlank DMA）

- HDMA1–5 (0xFF51–55) を実装。汎用転送と HBlank 同期転送の両モード。
- `GameBoy::step` の PPU モード遷移（HBlank 突入）にフックを差し込む。

### フェーズ 8: CGB ブートロム（任意）

- `core/src/bootrom.rs` は 0x100 固定。CGB ブートロムは 0x900。ブートロムを使う場合は
  サイズ可変化とアドレスマップ（0x0000–0x00FF と 0x0200–0x08FF の 2 領域、中央の
  0x0100–0x01FF はカートヘッダ）対応が必要。
- ブートロム無しで `apply_cgb_init` により起動できるようにしておけば、このフェーズは任意。

## テスト戦略

- **回帰**: フェーズ 1 完了時点で、既存の DMG ROM（`cpu_instrs.gb` 等）の表示・
  `cpu_instrs` の PASS が変わらないことを確認。
- **CGB 検証 ROM**: `cgb_sound` / `cgb-acid2`（PPU カラー・優先度の視覚検証）等の
  テスト ROM を用意し、フェーズごとに参照画像と突き合わせる。
- **teensy**: 既存の FPS / フレーム負荷オーバーレイでカラー化後の処理コスト退行を計測。

## リスク

- **性能退行**（特に teensy）: 属性・パレット参照でスキャンライン処理が重くなる。
  フェーズ 1 と 3 の後で必ず負荷計測する。
- **Display トレイト変更の破壊的波及**: `draw` シグネチャ変更は host / teensy 両方に及ぶ。
  フェーズ 1 で一括更新し、以降は PPU 内部だけの変更に閉じる設計にする。
- **メモリ増加**: teensy の静的確保が増える。`Ppu` を `static mut` 側へ寄せる等の配置検討。
- **優先度規則の複雑さ**: CGB の BG/OBJ 優先度は DMG と規則が異なり、cgb-acid2 等で
  丁寧に突き合わせる必要がある。
